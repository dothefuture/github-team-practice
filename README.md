## 5. Ошибка при Push без Pull

Один из участников попытался отправить изменения в удалённый репозиторий, не выполнив предварительно Pull.

В результате GitHub отклонил Push, так как удалённый репозиторий содержал более новые изменения.

### Решение

1. Выполнить Pull.
2. Получить актуальную версию проекта.
3. Объединить изменения.
4. Повторно выполнить Push.

### Проблема: забыли сделать Pull

Мы увидели, что если участник работает со старой версией проекта, Git может не разрешить отправить изменения сразу. Сначала необходимо получить актуальную версию проекта с GitHub, объединить изменения и только после этого выполнять Push.

## 10. Fetch и Pull

### Разница между Fetch и Pull

Fetch получает информацию о новых изменениях из удалённого репозитория, но не применяет их автоматически.

Pull получает изменения и сразу объединяет их с локальной версией проекта.

Таким образом:

* Fetch = проверить наличие изменений;
* Pull = получить и сразу применить изменения.

# Ответы на контрольные вопросы

### 1. Что такое репозиторий?

Репозиторий — это место хранения проекта и всей истории его изменений.

### 2. Чем локальный репозиторий отличается от удалённого?

Локальный находится на компьютере пользователя, удалённый — на сервере GitHub.

### 3. Что делает команда Pull?

Получает изменения из удалённого репозитория и объединяет их с локальной версией.

### 4. Что делает команда Push?

Отправляет локальные изменения в удалённый репозиторий.

### 5. Чем Fetch отличается от Pull?

Fetch только получает информацию об изменениях, Pull получает и сразу объединяет их.

### 6. Что такое ветка?

Ветка — это отдельная линия разработки внутри репозитория.

### 7. Почему не всегда удобно работать сразу в main?

Изменения разных участников могут конфликтовать и нарушать стабильность проекта.

### 8. Что такое Pull Request?

Pull Request — запрос на объединение изменений одной ветки с другой.

### 9. Зачем нужна проверка Pull Request другим участником?

Для поиска ошибок и контроля качества изменений.

### 10. Что такое merge conflict?

Ситуация, когда Git не может автоматически объединить изменения.

### 11. Почему возникает merge conflict?

Когда несколько участников изменяют один и тот же участок файла.

### 12. Как понять, какой вариант кода оставить при конфликте?

Необходимо проанализировать оба варианта и выбрать наиболее корректный либо объединить их.

### 13. Что будет, если забыть сделать Pull перед началом работы?

Могут возникнуть ошибки при Push и конфликты при объединении изменений.

### 14. Почему коммиты должны быть маленькими и понятными?

Так проще отслеживать изменения и находить ошибки.

### 15. Что было самым сложным в этой практической работе?

Разрешение конфликтов и работа с Pull Request.
